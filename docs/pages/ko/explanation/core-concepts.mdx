# 핵심 개념

구축을 시작하기에 네 가지 개념이면 충분합니다. 각 섹션에서는 개념을 정의하고, 보여주며, 전체 참조에 연결합니다.

## gas로서의 USDT0

트랜잭션 수수료는 이미 보유하고 거래하고 있는 자산인 USDT0으로 지불합니다. 자금을 조달하거나 관리해야 하는 두 번째 토큰은 없습니다.

USDT0은 기본 gas 자산(18진수, `address(x).balance`를 통해 읽음)이자 ERC-20 토큰(6진수, `USDT0.balanceOf(x)`를 통해 읽음)입니다. 두 인터페이스는 동일한 기본 잔액에서 작동하며, 프로토콜은 12자리 정밀도 차이를 자동으로 조정합니다.

```solidity
// 둘 다 동일한 잔액을 읽습니다.
uint256 native = address(user).balance;        // 18진수
uint256 erc20  = IERC20(USDT0).balanceOf(user); // 6진수
```

:::warning
잔액 조정은 예비 주소 `0x6D11e1A6BdCC974ebE1cA73CC2c1Ea3fDE624370`에서 추가 `Transfer` 이벤트를 발생시킵니다. `Transfer` 이벤트를 다시 재생하는 인덱서는 이 주소로의 전송 및 이 주소로부터의 전송을 필터링해야 합니다. 그렇지 않으면 잔액을 자동으로 이중 계산합니다.
:::

자세히 읽기: [gas로서의 USDT0](/ko/explanation/usdt-as-gas-token) · [Stable에서의 USDT0 동작](/ko/explanation/usdt0-behavior).

## 보장된 블록 공간

Stable은 각 블록 용량의 일부를 사전 할당된 엔터프라이즈 작업량에 예약합니다. 예약된 트래픽은 일반 트래픽이 혼잡한 경우에도 예측 가능한 대기 시간과 비용으로 정산됩니다. 이는 수수료 시장에서 경쟁하지 않습니다.

이 동작은 호출자 수준에서 투명합니다. 트랜잭션을 정상적인 방식으로 제출합니다. 할당은 등록된 계정에 대해 프로토콜 수준에서 적용됩니다.

자세히 읽기: [보장된 블록 공간](/ko/explanation/guaranteed-blockspace).

## USDT 전송 집계기

대량의 USDT0 전송은 MapReduce에서 영감을 받은 파이프라인을 사용하여 병렬로 일괄 처리되고 검증됩니다. 계정당 실패는 격리되므로 하나의 잘못된 전송이 배치를 중단시키지 않습니다.

호출자 측 전송 API는 변경되지 않습니다. 코드를 변경하지 않고도 처리량을 얻으면서 정상적인 방식으로 전송을 제출합니다.

자세히 읽기: [USDT 전송 집계기](/ko/explanation/usdt-transfer-aggregator).

## EVM 호환성

표준 EVM 도구는 변경 없이 작동합니다. EVM 수준에서 세 가지 동작이 이더리움과 다릅니다(위에서 다룬 gas로서의 USDT0은 네 번째입니다).

**단일 슬롯 완결성.** 트랜잭션은 블록에 포함되면 최종적입니다. 블록은 약 0.7초마다 생성됩니다.

**우선순위 팁 없음.** `maxPriorityFeePerGas`는 항상 무시됩니다. 유효 gas 가격은 프로토콜이 설정한 기본 수수료입니다.

```typescript
import { ethers } from "ethers";

const block = await provider.getBlock("latest");
const baseFee = block.baseFeePerGas;

const tx = await wallet.sendTransaction({
  to: "0xRecipientAddress",
  value: ethers.parseEther("0.1"),
  maxFeePerGas: baseFee * 2n,        // 안전 여유로 기본 수수료의 2배
  maxPriorityFeePerGas: 0n,          // Stable에서는 항상 0
});

await tx.wait();
console.log("Included at gas price:", tx.gasPrice?.toString());
```

```text
Include at gas price: 1000000000
```

**이중 역할 USDT0, 포팅 위험.** 이더리움에서 포팅된 계약은 기본 잔액을 미러링해서는 안 되며, `address(0)` 전송을 거부해야 하며, 주소 재사용 감지를 위해 `EXTCODEHASH`에 의존해서는 안 됩니다.

:::warning
내부 변수에 기본 잔액을 미러링하는 계약을 Stable에 포팅하는 것은 안전하지 않습니다. 외부 `USDT0.transferFrom` 호출은 계약의 기본 잔액을 계약 코드를 호출하지 않고 소진할 수 있습니다. 전송 시점에 항상 `address(this).balance`로 지급 능력을 확인하세요.
:::
